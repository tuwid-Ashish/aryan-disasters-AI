# Student Testing Guide — Disaster Response SaaS (With AI)

This guide helps students run, test, and evaluate the Disaster Response SaaS project, including the AI features (explainability, disaster summary, and match suggestions).

## Goals
- Run the app locally (server + client)
- Create test data (disaster, requests, resources)
- Exercise AI features and validate outputs
- Run minimal automated checks included with the project

## Prerequisites
- Node.js 18+ installed
- MongoDB running locally (or a cloud MongoDB URI)
- Basic git familiarity

## Setup

1. Clone the repo and install dependencies:

```bash
git clone <repo-url>
cd "aryan disasters AI"
cd server && npm install
cd ../client && npm install
```

2. Create a `.env` file in `server/` with the required variables (example):

```env
MONGODB_URI=mongodb://127.0.0.1:27017/adai-dev
JWT_SECRET=some_long_secret
CLIENT_URL=http://localhost:5173
# Optional LLM keys (one of these for richer AI responses):
# GEMINI_API_KEY=your_gemini_key
# GROQ_API_KEY=your_groq_key
# Optionally set provider: LLM_PROVIDER=gemini
```

3. (Optional) If you don't set an LLM key, the system will still work using the deterministic local fallback for AI responses.

## Running the App

Open two terminals.

Server:
```bash
cd server
npm run dev
```

Client:
```bash
cd client
npm run dev
```

Open the client at `http://localhost:5173`, register an account, and promote a user to the `admin` role in the database (or use provided seed data if available).

## Creating Test Data (quick steps)

1. As Admin → Disasters → Create a disaster (title, region, severity = high, status = active).
2. As Beneficiary → Create a request for that disaster (category: water/medicine/food/shelter), then as Admin → Verify the request.
3. As Donor → Create a resource matching that category, and as Admin → Verify the resource.

Ensure at least one verified request and one verified resource exist — AI matching and suggestions rely on verified items.

## Where to Find AI Features (Admin UI)

- Analytics: `Admin → Analytics` — shows an AI-generated disaster summary in the "System summary" section.
- Verify Requests: `Admin → Verify Requests` — click **Explain AI** on a request to view a modal with the AI rationale and priority score.
- Allocation Center: `Admin → Allocation Center` — view AI match suggestions (request → resource) and the LLM rationale above the table.

Notes:
- If an LLM key is configured, explanations and summaries will be generated by the external API (Gemini/Groq).
- If no key is configured, the UI still shows deterministic fallback text so functionality is testable without API access.

## AI Endpoints (for advanced testing)

- `POST /api/ai/explain-priority` — expects a request object with `category`, `urgencyLevel`, etc.
- `POST /api/ai/disaster-summary` — expects a disaster payload: `title`, `region`, and optional context.
- `POST /api/ai/match-resources` — expects `requests: []` and `resources: []` arrays; returns suggestions and LLM rationale.

Use these with `curl` or Postman for manual verification. Example:

```bash
curl -X POST http://localhost:5000/api/ai/disaster-summary \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"title":"Flood Zone A","region":"Mumbai"}'
```

## Running Tests

Server contains minimal tests for AI logic. Run:

```bash
cd server
npm test
```

These tests check that the AI service falls back gracefully when no external LLM key is present and that deterministic matching works.

## Demo Script (5–10 minutes)

1. Start server + client.
2. Create a disaster named `Flood Demo`.
3. Create and verify a beneficiary request for `medicine` with `urgencyLevel: critical`.
4. Create and verify a donor resource `medicine` with adequate quantity.
5. Admin → Verify Requests → Click **Explain AI** on the request → observe priority score + explanation.
6. Admin → Allocation Center → observe suggested match and the LLM rationale.
7. Admin → Analytics → confirm AI disaster summary appears.

## Evaluation Checklist (for instructors)

- App runs locally without errors.
- Admin can create a disaster and verify requests/resources.
- AI features are accessible and return text (LLM or fallback).
- `npm test` in `server/` passes.

## Troubleshooting

- If AI responses are always the fallback text, verify the `GEMINI_API_KEY` or `GROQ_API_KEY` in `server/.env` and restart the server.
- If UI buttons don't appear, ensure you're logged in as an admin and that the page has loaded fully.
- Check server logs (server terminal) for LLM errors and rate-limit messages.

## Notes on Ethics & Safety

AI outputs are suggestions only — all allocations must be approved by an Admin. Avoid using AI outputs as the sole decision source for real-world relief operations.

Gemini responses in this project are generated through Google's official Gemini SDK, so if the API key is valid the app will use the official SDK path instead of a custom HTTP call.
